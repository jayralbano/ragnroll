# Keypoints

Source: `2026-03-04 23-51-53 long meeting big change.mp4`

1. **Parted ways with Amar, major company changes ahead** — The company has parted ways with Amar, and leadership (speaker + Asim) has been quietly evaluating the massive AI-driven shift in technology. The last 12 months have been the biggest transformation in tech in 50 years, and the current 2019-era codebase, development workflows (daily scrums, ticket-based shipping), and infrastructure are now obsolete. If the company doesn't radically adapt — not in a year, but now — it won't survive. The old ways are costing too much time and money, and rebuilding the product from scratch would look completely different today.

2. **The plan — maintain legacy, build API, ship companion products fast** — The legacy platform stays in maintenance mode (Adam stepping in to manage daily scrums/tickets replacing Amar). The first big rock is extracting membership.io into APIs (content library, payments, subscriptions, member management, content permissions) since nothing can move forward without them. Then build modern companion apps/services around those APIs — including agentic experiences (conversational UI, auto-reports, AI-built hubs). Development shifts from front-end/back-end siloes to single-person or pod-based end-to-end shipping. Most time will be spent on requirements, planning, and vision — not coding. Features that used to take a month should ship in a day.

3. **Everyone becomes an engineer — roles merge, build your own toolkits** — No more siloed roles (QA, front-end, back-end). Everyone is now an engineer who owns end-to-end delivery. QA people (Claudiu, Sherry) should be building automated testing pipelines and tools, not manually clicking around. Each person builds their own toolkit of tools-within-tools that can be pointed at any project. Planning and specs become 90% of the work — Asim describes Jayr's notification project as the model: do research with AI agents, present a proposal, get high-level sign-off, then ship it. Mihai confirms devs can now have a full idea, build an MVP in hours, and demo it.

4. **Planning is the new coding — vibe coding vs real engineering** — Asim addresses the "vibe coding" trend: YouTube/TikTok marketers building apps with AI give a false impression. Real engineering experience (database indexing, security, infrastructure) is now the hottest commodity because it feeds into planning context. AI models have perfected code generation — syntax is memorized — but the quality of output is 100% dependent on the quality of context and planning you provide. "Shitty inputs = shitty outputs." The old sprint planning with story points was an archaic version of this. Now your engineering experience IS the planning, and coding is just execution.

5. **Jayr's notification workflow as the new development model** — Asim walks through Jayr's notification project as the template for how work should be done going forward. Instead of leadership interviewing support, creating Jira tickets, and handing specs to devs: Jayr launched parallel AI agents to (1) read all HubSpot support tickets about notifications, (2) reverse-engineer the existing notification system, (3) analyze all related Jira tickets. The agents synthesized findings into a phased rollout plan. Jayr brought that back for approval, leadership reviewed with their own Claude agents for validation, then Jayr got the green light to execute. QA still matters — the team needs to build automated QA systems and pipelines, not manual testing.

6. **QA is the next bottleneck — automate or die** — As AI-driven development speeds up shipping, QA will become the critical bottleneck. The legacy system had no unit tests for the first 3 years. Now the pipeline must be: unit tests written in specs first, then agent-generated code, then automated smoke testing (using AI vision for cross-browser testing). Manual QA by Claudiu/Jake/Sherry clicking around must be replaced by overnight automated runs. Asim admits he doesn't have a full answer yet — the team needs to research how companies like Stripe handle QA in agentic workflows. Everyone needs to treat this as a shared team problem, not just the QA team's burden.

7. **Jira stays for now but tools must be AI-agent friendly** — Claudiu asks if Jira will remain for tracking. Asim says yes for now (support and processes depend on it), but Jira isn't ideal for AI agents — reading one ticket filled 90% of Claude's context window. Asim moved to GitHub Issues for personal projects and even built a custom markdown-based issue tracker via Claude Code in 20 minutes. Key insight from Marcus: tickets now serve as shared source of truth for AI agents to ingest, not just human task tracking. Products that aren't AI-agent compatible (including membership.io itself) will die a slow death — this is also a business hint for their own product direction.

8. **AI code review, language-agnostic dev, senior experience is king** — Marius argues code review should be done by AI too, since devs will be building in languages they don't know (Swift, Java, etc.) and human review would slow things down. Move toward framework/language-agnostic development with AI-enforced quality rules and automated tests. Asim agrees but emphasizes: senior engineering experience is the real differentiator — knowing about database indexing, row-level security, performance at scale. These must be in your plans BEFORE agents code. Vibe coders skip this and get hacked or collapse at 100 users. The team should build personal "skills" and workflow harnesses (like the superpowers brainstorm-plan-execute pipeline) to enforce quality. Burn tokens freely — they're cheap compared to human time.
